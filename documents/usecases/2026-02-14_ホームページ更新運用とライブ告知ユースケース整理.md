# 1212ホームページ更新運用 ユースケース整理

## 背景・目的

### なぜこの整理が必要か

- ホームページ更新と告知発信の作業が分断されており、更新漏れ・告知漏れが起きやすい
- 今後の運用を「本人1名で継続可能」な形にするため、業務フローを明文化する必要がある

### 現状の課題

- 管理画面で編集しても、公開反映までの手動作業が残る
- ライブ情報とX告知が別作業のため、タイミングがずれる
- スマホだけで完結しづらい

### ゴール

- 管理画面からライブ情報を登録すると、公開反映と告知発信が一連で完了する
- 投稿失敗時も、管理画面上で再実行できる
- 本人のみが安全に更新できる

## 目指す構造

### システム構成

- 本人専用のWeb管理画面で更新を実施
- ライブ登録時に告知作成・X投稿まで同一運用で完了
- 投稿結果を管理画面で確認・再実行できる

### 各アクターに必要な機能

#### 管理者

| 機能カテゴリ | やりたいこと | 優先度 | 備考 |
|-------------|-------------|--------|------|
| 認証 | 本人のみログインして編集 | 高 | 不正編集防止 |
| ライブ登録 | 日付/会場/詳細/導線をスマホで登録 | 高 | 低負荷入力 |
| 告知投稿 | 登録内容を整形してXへ投稿 | 高 | 初期はXのみ |
| 障害対応 | 投稿失敗時の再実行 | 高 | 運用継続性 |
| 履歴確認 | いつ何を更新・投稿したか確認 | 中 | トラブル時の確認 |

#### 閲覧者

| 機能カテゴリ | やりたいこと | 優先度 | 備考 |
|-------------|-------------|--------|------|
| 情報取得 | 最新ライブ情報をホームページで確認 | 高 | 反映遅延を最小化 |
| 告知取得 | Xでライブ告知を確認 | 高 | 拡散導線 |

## ユースケース詳細

### UC-01: ライブ情報を登録して告知まで完了する

| 項目 | 内容 |
|------|------|
| アクター | 管理者 |
| 目的 | ライブ情報登録とX告知を同時に完了する |
| 事前条件 | 管理者が認証済みである |
| 基本フロー | 1. 管理画面でライブ情報を入力する<br>2. 入力内容を確認して保存する<br>3. 告知文を確認して投稿実行する<br>4. 公開反映と投稿完了を確認する |
| 代替フロー | 投稿に失敗した場合、原因表示後に再実行する |
| 事後条件 | ホームページ情報とX告知が整合した状態になる |

### UC-02: ライブ情報を修正し告知を更新する

| 項目 | 内容 |
|------|------|
| アクター | 管理者 |
| 目的 | 登録済みライブ情報の誤りを修正し、必要に応じて再告知する |
| 事前条件 | 対象ライブが登録済みである |
| 基本フロー | 1. 対象ライブ情報を選択する<br>2. 内容を修正して保存する<br>3. 必要時のみ告知を再投稿する |
| 代替フロー | 再投稿不要の場合、公開反映のみで完了する |
| 事後条件 | 公開情報が最新化され、告知の整合性が保たれる |
